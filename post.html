In het kader van het tentamen voor het vak ‘Design for interaction: Models & processes’ (uit de V1) heb ik een globale samenvatting geschreven. Het is dus een hele kleine samenvatting om overzicht te geven, geen compleet pakket.

In mijn samenvatting kan je hoofdstukken 2, 7, 8, 9, 10, 11, 12, 14 en 26 hieronder vinden. Dit zijn alle verplichte hoofdstukken, de bonus hoofdstukken zitten er niet bij. Onderaan vindt je een opsomming van alle design principles.

Chapter 2: Mentale modellen

Een systeem kent 3 modellen:

  • Het implementatie model
    Dit is een technisch model die omschrijft hoe het model technisch in elkaar steekt.

  • Het mentale model
    Dit is een vaag model die omschrijft hoe de gebruiker denkt dat het product werkt. De nadruk ligt op de interactie tussen de gebruiker en het systeem.

  • Het representatie model
    Dit is het model waar je je tijdens het ID proces mee bezig houdt. Het is een combinatie van de twee bovenstaande. Hoe meer hij naar het mentale model neigt hoe beter.

Chapter 7: The Framework

Dit is het logische vervolg op de materie uit het vak ‘Research for design’.

Het Design framework kent twee onderdelen:

  • Het Design framework
  • Het Interaction framework

Deze onderdelen kennen beide stappen (niet perse chronologisch).

Het Design framework

  1. Form factor, posture and input models
    Wat wordt het voor product? Wat is de context? Op welke manier vindt interactie plaats?

  2. Functional and data elements
    Wat voor content zit er in het systeem? Wat voor ID functie’s kent het systeem (dit laatste op het niveau van ‘informatie architetuur’)?

  3. Group functions
    Maak een hierarchie van al je ID functie’s.

  4. Sketch interaction framework
    Schets (visualiseer) je ID -> Interaction design framework

  5. Key path scenario
    Een taak georienteerd scenario van je hoofdfunctie’s (ID) in een ideale situatie.

  6. Check designs with a validation scenario
    Test! Ook via usability testing.

Het Visual framework

  1. Define the visual design framework
    Doe onderzoek naar bestaande onderzoeken mbt. visual design.

  2. Apply the chosen style to the screen archetype
    Pas het visual design toe op het medium van je systeem.

Chapter 8: Design principles and Patterns

Hoe kom je tot beslissingen in je proces (in alle stappen van het proces)?

Principles

Een generale richtlijn. die gaat over de achterliggende gedachte van de keuze: waarom kiezen we voor A?

Design values:

  • Etisch
  • Nutvol
  • Pragmatisch (praktisch)
  • Elegant: De meest simpel werkende oplossing

Patterns

Oplossing voor een specifiek probleem. Altijd concreet in plaats van abstract. Nooit een kant en klare oplossing!

Typen ID Patronen:

  • Postural: aanzien product tegenover gebruiker
  • Structural
  • Behavioral: Lower level patters

Chapter 9: Platform and Posture

Platform

Een combinatie van de hardware en software waar je systeem op draait. Zowel het technische als het interactie aspect.

  • voorbeelden: desktop software, handheld, professional devices

posture

De houding van het systeem tegenover de gebruiker. Er zijn drie typen postures: sovereign, transient en daemonic.

Platform en Posture bij desktop applicatie’s

Het platform van een desktop applicatie is meer dan alleen ‘desktop applicatie’. Het betreft ook meer techniek zoals: OS, database, internet en hardware specificatie’s.

posture van desktop applicatie’s:

  1. Sovereign posture

    wanneer? als een gebruiker voor lange tijd met je systeem bezig is.
    voorbeelden: text editing, mail, etc.
    kenmerken: fullscreen, learning curve, interface gestructureerd volgens veel gebruikte functie’s.

  2. Transient posture

    wanneer? zodra je systeem in een kleine snelle taak faciliteert.
    voorbeelden: rekenmachine, windows explorer, weather widget, etc.
    kenmerken: zo klein mogelijk / movable, wordt vaak aangeroepen door een Sovereign posture.

  3. Daemonic posture

    wanneer? bij een service of proces die op de achtergrond draait.
    voorbeelden: drivers, batterij symbool van een laptop, netwerk symbool computer.
    kenmerken: minimale interface, interface geeft vaak de status aan.

Platform en Posture bij websites

Er zijn verschillende typen websites die elk passen bij een andere posture.

  • Informational websites
    posture: balans tussen content en een lage learning curve.

  • Transanctional websites
    Dit zijn websites die meer interactie hebben dan alleen het navigeren en verschaffen van content. posture: elegant, schijnt snel te zijn*.
    *Dit is meer afhankelijk van of de gebruiker zijn doel bereikt dan de daadwerkelijke tijd.

  • Web applications
    Denk aan online publishing tools, enterprise software, etc.
    posture: Dit gaat heel erg richting het Sovereign posture van desktop applicatie’s. Houdt wel rekening met de limitatie’s van het platform.
    Je kan web apps ook goed inzetten om problemen mbt. een transient posture op te lossen, de gebruiker hoeft een web app namelijk niet te installeren.

Platform en Posture in overige gevallen

  • Zie je systeem niet als een PC
  • Integreer hardware met software
  • Context = key
  • Customize for platform

Chapter 10: Orchestration and Flow

Zodra je gebruiker geconcentreerd bezig is met je systeem en niet snel afgeleid raakt, bevindt hij zich in een staat. Deze staat heet de flow.

In de flow is een gebruiker extreem productief. Het is dus belangrijk om hem in de flow te houden. Over het algemeen geldt: hoe minder interface hoe meer flow.

Transpirantie slaat op hoe automatisch een interface gaat. Hoe transpiranter hoe minder de gebruiker doorheeft dat hij bezig is met je systeem. Hoe transpiranter hoe meer flow.

Orchestration is de harmonie in ID:

Hier zijn geen vaste regels voor maar wel richtlijnen:

  • Volg het mentaal model
  • Less is more
  • Geef modeless feedback
  • Design voor waarschijnlijk, geef ruimte voor misschien
  • Bied keuzes
  • Optimaliseer voor ‘responsiveness’

Chapter 11: Eliminating Excise

Grote taken kan je onderverdelen in kleinere taken. Zodra zo’n kleine taak niet direct bijdraagt aan de hoofdtaak is dit excise, houd excise altijd zo klein mogelijk.

Als je iets op een website wil kopen is de ‘vergelijk taak’ excise en alles mbt. het winkelwagentje en afrekenen niet. Je hebt excise taken dus wel nodig om te slagen in je hoofddoel.

Voor een hoop taken kan je je doel sneller bereiken met een CLI (command line interface) dan met een GUI (graphical user interface). Echter moeten gebruikers van CLI wel alle commando’s van te voren al weten, je keuze hangt af van de doelgroep en hoe vaak ze iets gebruiken.

Alles wat je systeem zelf had kunnen uitvogelen maar het toch vroeg is excise (denk hierbij aan het vragen om toestemming).

Zelfs de google resultaten pagina is excise omdat het niet direct antwoord geeft op je vraag (technisch nog niet haalbaar).

Chapter 12: Designing good behaviour

Als een systeem zich als een ‘likable persoon’ gedraagt vinden mensen het systeem ook echt leuker.

De computer doet het werk, de gebruiker doet het denken.

Good behaviour kan je bereiken door ervoor te zorgen dat je systeem zich de volgende dingen is:

  • Considerate
  • Polite
  • Takes interest -> remembering
  • deferential -> submitting to the user
  • Anticipate
  • Informational
  • Perceptive -> oplettend
  • Fail gracefully

Daarnaast moet je systeem ook smart zijn. Geen domme vragen stellen en dingen onthouden spelen hier een grote rol.

If it’s worth the user entering, it’s worth the system remembering.

Chapter 14: Visual Interface Design

VID (Visual Interface Design) gaat over het communiceren van je systeem aan je gebruiker. Dit is veel belangrijker dan hitting it with the pretty stick.

Het design van je interface draagt bij aan de herkenbaarheid en dus de visceral state van de gebruiker.

VID representeert ook info en gedrag, dit moet altijd gerelateerd zijn aan het doel van de info of het gedrag. Het belangrijkste aan VID is de gedachtegang achter de communicatie en niet zozeer de vorm. Maak elementen visueel anders als hun gedrag / functie ook anders is.

The building blocks of VID:

  • Shape
  • Size
  • Value
  • Hue -> draws attention (or not)
  • Orientation
  • Texture
  • Position

Visual Patterns hebben een hoofdrol in VID omdat de gebruiker zo de hele interface snel kan doorgronden.

Visually show what, textually show which.

Chapter 26: Designing for different Needs

Command Vectors

Dit zijn technieken waarmee gebruikers instructie’s aan het programma kunnen geven. Denk aan:

  • Knoppen
  • Dropdown
  • Toolbar controls
  • Keyboard accelerators (shortkeys)

Een Immediate vector is een vector waar de gebruiker gelijk na de actie het resultaat ziet. Bij een Pedagogic vector is er minimaal een tussenstap (denk aan een menu).

Working Set

Als een gebruiker je systeem gebruikt onthoudt hij onderbewust de commands. Al deze commands samen zijn zijn Working set. Dit is dus per gebruiker uniek.

The minimal working set staat voor de commando’s die je persona vaak nodig heeft bij het doorlopen van je scenario’s.

Information in the world: info in de interface of in de omgeving van de gebruiker.
Information in the head: info in het geheugen van de gebruiker.

Dit laatste is veel sneller, echter moet de gebruiker het wel weten / onthouden.

World vector: commando aanwezig in de interface.
Head vector: commando onthouden door de gebruiker.

Beginning en Intermediate gebruikers hebben de natuurlijke behoefte om hun working set uit te breiden zodat ze sneller naar hun doel kunnen, dit doen ze door nieuwe Head vectors te leren.
